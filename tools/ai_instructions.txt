* Репозиторий на GitHub: [Aleksey095599/alligator-market](https://github.com/Aleksey095599/alligator-market).
* Backend – Spring Boot 4.
* Frontend – Angular (TypeScript).
* Разработка ведется в IntelliJ IDEA Ultimate.

2. --- Мои общие ценности в проектировании и написании кода ---
* Акцент на качественную DDD архитектуру - приоритет продуманных доменных моделей и бизнес логики.
* Логичная структура папок ("все по полочкам").
* Продуманная "простота" обеспечивает надежность, "все гениальное просто".


3. --- Триггерное слово «@задача» ---
Если в сообщении пользователя есть слоао «@задача» и приводится описание этой задачи, то следовать пунктам А) и Б) данного раздела.
А) Инструкция для размышления модели ChatGPT:
* "Решение согласно лучшим практикам": проанализируйте лучшие подходы в крупных успешных проектах, решающих похожие задачи.
* "Простое изящное решение": Найдите простое и изящное решение, полностью удовлетворяющее требованиям.
* "Итоговое решение": добавьте в архитектуру "простого изящного решения" точки роста к "решению согласно лучшим практикам".
Б) Формат ответа:
* Описать "простое изящное решение".
* Описать "итоговое решение" и объяснить почему оно перспективное.

4. --- Правила оформления ---
Общий принцип: оформлять код согласно лучшим мировым практикам без шума.
Основные правила оформления:
* Добавляйте комментарии (на русском языке, кроме устоявшихся терминов) над важными для логики кода объектами, чтобы пояснить их назначение.
* Добавляйте краткие комментарии (на русском языке, кроме устоявшихся терминов) в коде там, где это важно для понимания логической цепочки.
* Допускается применять в комментариях стрелочки или иные символы для наглядности, если это уместно и считается хорошей практикой.
* Текст логов, сообщения об ошибках, внешние сообщения и т.п. оформляйте на английском языке согласно лучшим стандартам.

5. --- Commit generation ---
Общий принцип: оформлять согласно принятым стандартам, но кратко и без шума.

6. --- Конвенция config (Spring wiring) ---
* config/** содержит только wiring Spring-бинов: @Configuration, @Bean, @Import, @EnableConfigurationProperties и вспомогательные context-level infrastructure beans.
* Любой *Config самодостаточен: если его бинам нужны другие бины, зависимости подключаются явно через @Import.
* Классы внешних настроек называются *Properties, лежат вне config/** и подключаются явно через @EnableConfigurationProperties(...).
* Бизнес-логика, доменные модели, application services, repository adapters и mapper’ы не создаются через stereotype-аннотации (@Component, @Service, и т.п.); значимые feature-бины собираются явно через config/**.
* Имена бинов не дублируются строками: использовать константы BEAN_... в @Bean и @Qualifier

7. --- Каноническая архитектура фичи ---
* Каждая фича проектируется как самодостаточный vertical slice вокруг одного доменного смысла и одного основного aggregate root.
* Фича должна быть разделена на пять независимых ролей: domain, application, api, infrastructure/persistence, config/wiring.
* Domain содержит только предметную модель: aggregate root, value objects, внутренние элементы агрегата, domain services и repository ports. Домен не зависит от Spring, HTTP, SQL, JPA, jOOQ и UI.
* Application содержит только use case orchestration: command/query сценарии, application ports, application exceptions. Этот слой координирует домен и внешние зависимости, но не содержит SQL, HTTP-деталей и framework-магии.
* API содержит только transport-адаптеры: controllers, request/response DTO, feature-specific exception handling, API mappers. API не должен содержать бизнес-логику и persistence-логику.
* Infrastructure / persistence содержит только реализации портов: jOOQ/JPA/SQL adapters, интеграции с внешними системами, query adapters. Этот слой не должен принимать архитектурные решения за domain/application.
* Config / wiring содержит только явную сборку Spring-бинов. Все значимые бины фичи собираются явно через @Configuration, @Bean, @Import, @Qualifier, BEAN_... constants. Не полагаться на скрытый component scanning для application/persistence/business beans.
* Внутри одного aggregate root должен быть один основной repository port. Для внутренних элементов агрегата не создавать отдельные доменные репозитории, если они не являются самостоятельными aggregate root.
* Feature сначала проектируется через whole-aggregate lifecycle: создать агрегат целиком, заменить агрегат целиком, удалить агрегат целиком, прочитать агрегат целиком. Более узкие операции добавляются только после стабилизации базового жизненного цикла.
* Query/read-side допускается проектировать отдельно от write-side: UI, таблицы, витрины, списки и dashboard-представления не должны диктовать форму доменной модели.
* Каждый слой обязан иметь одну роль и одну причину для изменения. Если класс одновременно “и use case, и SQL, и HTTP mapper”, значит архитектурная граница нарушена.
* Названия пакетов и классов должны прямо отражать роль объекта в архитектуре. Структура фичи должна быть очевидна по дереву папок без необходимости читать код.
* Простота считается преимуществом только тогда, когда она не размывает границы ответственности. Предпочтение отдаётся простому, явному и расширяемому решению.

8. --- Быстрая самопроверка фичи ---
Перед завершением фичи проверять:
* Можно ли понять структуру фичи по папкам без чтения реализации?
* Есть ли у aggregate root одна явная точка входа через repository port?
* Не протекают ли SQL/HTTP/Spring детали в domain?
* Не смешаны ли use case orchestration и persistence?
* Не подгоняется ли домен под UI-таблицу или DTO?
* Можно ли расширять фичу новыми сценариями без перестройки её основы?
